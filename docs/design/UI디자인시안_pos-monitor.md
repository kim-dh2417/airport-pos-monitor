# 공항 POS 시스템 UI 디자인 시안 초안

## 1. 문서 개요

- 기준 문서:
  - `docs/design/와이어프레임_pos-monitor.md`
  - `docs/design/화면설계서_pos-monitor.md`
- 대상 범위:
  - `메인 대시보드`
  - `매장 상세`
- 목적:
  - 운영형 대시보드에 맞는 시각 방향, 색상 체계, 타이포그래피, 핵심 컴포넌트 스타일을 정의한다.

## 2. 디자인 컨셉

### 컨셉명

`Airport Control Board`

### 방향성

- 공항 운영실의 정밀함과 대시보드형 모니터링 화면의 긴장감을 결합한다.
- 과도하게 어두운 관제실 화면이 아니라, 밝은 바탕 위에 강한 정보 대비를 주는 운영형 UI로 간다.
- 상태 정보는 감성보다 판독성을 우선한다.

### 키워드

- 명확함
- 운영성
- 감사 가능성
- 데이터 중심
- 절제된 긴장감

## 3. 시각 시스템

### 3.1 색상 방향

| 용도 | 값 | 설명 |
| --- | --- | --- |
| `bg.canvas` | `#F4F7FB` | 전체 배경 |
| `bg.surface` | `#FFFFFF` | 카드/패널 배경 |
| `bg.header` | `#0F2747` | 글로벌 헤더 배경 |
| `bg.subtle` | `#EAF1F8` | 필터 바, 안내 박스 |
| `text.primary` | `#17263C` | 기본 본문 |
| `text.secondary` | `#5B6B81` | 보조 텍스트 |
| `line.default` | `#D7E0EA` | 구분선 |
| `accent.primary` | `#0D6EFD` | 핵심 액션 |
| `status.wait` | `#7E8795` | 대기 |
| `status.normal` | `#1F8A5B` | 정상 |
| `status.review` | `#D89216` | 확인 필요 |
| `status.alert` | `#D6453D` | 이상 의심 |

### 3.2 타이포그래피

- 기본 UI 폰트:
  - `SUIT Variable`
- 숫자와 표, 차트 라벨 강조 폰트:
  - `IBM Plex Sans KR`

### 3.3 타입 스케일

| 요소 | 스타일 |
| --- | --- |
| 페이지 제목 | 28px / 700 |
| 카드 수치 | 30px / 700 |
| 섹션 제목 | 18px / 700 |
| 표 본문 | 14px / 500 |
| 보조 라벨 | 12px / 600 |

### 3.4 코너와 그림자

- 카드 라운드: `16px`
- 필터/입력 라운드: `12px`
- 그림자: 강한 드롭섀도 대신 얕은 분리감
- 상태 강조는 그림자보다 `좌측 컬러 바` 또는 `배지`로 처리

## 4. 핵심 컴포넌트 스타일

### 4.1 글로벌 헤더

- 깊은 남색 배경
- 좌측 서비스명
- 우측에 기준 월, 사용자 정보, 로그아웃
- 높이는 넉넉하게 두고 답답하지 않게 구성

### 4.2 상태 배지

- 텍스트만 있는 배지 대신 `컬러 점 + 텍스트 + 옅은 배경` 조합
- 예:
  - `대기`: 회색 점 + 회색 배경
  - `정상`: 초록 점 + 연초록 배경
  - `확인 필요`: 주황 점 + 연주황 배경
  - `이상 의심`: 빨강 점 + 연빨강 배경

### 4.3 요약 카드

- 카드 상단에 라벨
- 중앙에 큰 수치
- 하단에 짧은 설명 또는 전월 대비 변화 문구
- 카드 상단 좌측에 얇은 컬러 바를 넣어 상태별 연계를 줄 수 있다.

### 4.4 위험도 테이블

- 흰 배경 카드 위에 배치
- 헤더는 옅은 회청색 배경
- 행 hover 시 연한 파랑 배경 강조
- `이상 의심` 행은 전체 배경을 물들이지 않고 좌측 4px 컬러 스트립으로만 강조
- 숫자 셀은 `IBM Plex Sans KR` 사용

### 4.5 판단 근거 패널

- 일반 정보 카드보다 조금 더 강한 배경 대비 사용
- 핵심 사유는 번호 대신 불릿과 키워드 하이라이트로 표현
- 예외 가능성은 본문보다 작은 보조 태그로 묶는다.

### 4.6 차트

- 기본 라인 1: 신고금액 `#0D6EFD`
- 기본 라인 2: 비교 기준값 `#1F8A5B`
- 위험 구간 마커: `#D6453D`
- 축선과 그리드는 최대한 옅게 처리

### 4.7 일별 매출 캘린더

- 달력 셀은 단순 숫자 박스가 아니라 작은 데이터 카드처럼 사용
- 셀 상단:
  - 일자
- 셀 본문:
  - `P` POS
  - `V` VAN/PG
  - `D` 차이
- 이상 징후 셀은 테두리 강조 방식 사용

## 5. 화면별 UI 시안 방향

### 5.1 메인 대시보드

#### 톤

- 관리자 첫 화면답게 가장 정돈된 화면
- 상단은 안정적, 하단 테이블에서 긴장감이 생기는 구조

#### 레이아웃 방향

1. 상단 헤더 밴드
   - 좌측 제목과 설명
   - 우측 액션 버튼 2개

2. 요약 카드 5개
   - 한 줄 배치
   - `이상 의심` 카드만 수치 강조 색을 더 강하게 사용

3. 필터 바
   - 흰 배경 카드 대신 옅은 블루그레이 패널
   - 검색창은 좌측, 선택형 필터는 우측

4. 위험도 테이블
   - 행 높이를 너무 낮추지 않는다.
   - `차이금액`, `차이율`, `상태`가 한눈에 읽히게 시각적 대비를 준다.

5. 하단 운영 정보
   - 경고/재비교/누락을 작은 보조 카드 2~3개로 정리

#### 시안 메모

- 대시보드는 색이 많아지기 쉬우므로 상태색 외 일반 UI 색은 절제한다.
- 핵심 강조는 `파랑`, 상태 강조는 `초록/주황/빨강`으로 역할을 분리한다.

### 5.2 매장 상세

#### 톤

- 대시보드보다 더 분석적이고 깊이 있는 화면
- 숫자와 근거가 먼저, 그래프와 패턴은 그다음

#### 레이아웃 방향

1. 상단 요약 존
   - 상태 배지를 제목 옆에 두고 즉시 인지되게 처리
   - 기준 월과 POS 코드 등 메타 정보는 보조 톤으로 정렬

2. 핵심 비교 카드
   - 5개 카드가 한 줄 또는 3+2 배치
   - `차이금액`, `차이율` 카드는 더 큰 숫자로 강조

3. 판단 근거 패널
   - 밝은 백색 카드가 아니라 옅은 남색 또는 회청색 틴트를 깐 패널
   - "왜 이상 의심인지"가 첫 스크롤 안에 보이게 한다.

4. 월별 추이 그래프
   - 가로 폭을 넉넉히 사용
   - 금월과 전월은 점 마커와 주석으로 강조

5. 일별 캘린더 + 선택 일자 상세
   - 좌우 분할 배치
   - 캘린더에서 날짜를 누르면 우측 세부정보가 즉시 갱신

6. 외부 참고 데이터
   - 별도 카드 그룹
   - 연동 전에는 회색 아웃라인 카드로 빈 상태를 디자인한다.

#### 시안 메모

- 상세 화면은 시각적으로 복잡해질 수 있으니 한 카드 안에 지표를 과하게 넣지 않는다.
- 그래프와 캘린더 사이에 충분한 간격을 두어 서로 경쟁하지 않게 한다.

## 6. 상태 표현 가이드

| 상태 | 배지 | 행/카드 강조 | 설명 톤 |
| --- | --- | --- | --- |
| `대기` | 회색 배지 | 회색 아웃라인 | 중립 |
| `정상` | 초록 배지 | 연초록 상단 바 | 안정 |
| `확인 필요` | 주황 배지 | 연주황 포인트 | 주의 |
| `이상 의심` | 빨강 배지 | 빨강 좌측 바 | 강한 경고 |

중요:

- 상태는 색만으로 구분하지 않는다.
- `대기`는 경고처럼 보이면 안 된다.
- `확인 필요`와 `이상 의심`은 명도 차이보다 의미 차이가 먼저 읽혀야 한다.

## 7. 반응형 방향

- 데스크톱:
  - 카드 + 테이블 + 차트 조합
- 태블릿:
  - 2열 카드 레이아웃
  - 테이블 일부 컬럼 축약
- 모바일:
  - 대시보드 테이블을 카드 리스트로 전환
  - 상세 화면은 요약 카드와 판단 근거를 우선 노출
  - 캘린더는 정보량을 줄이고 선택 일자 상세를 아래에 크게 배치

## 8. UI 문구 톤

- 단정하고 운영 친화적인 문구를 쓴다.
- 감정적 표현보다 사실 중심 표현을 사용한다.

예시:

- `이상 의심 매장 4건`
- `필수 데이터 미접수로 비교가 보류되었습니다`
- `전월 대비 차이율이 임계치를 초과했습니다`
- `외부 참고 데이터는 아직 연동되지 않았습니다`

## 9. 다음 단계 제안

- 다음 라운드에서 `로그인`, `수기 등록`, `엑셀 업로드` 화면까지 같은 톤으로 확장한다.
- 구현 단계에 들어가기 전, 이 문서를 바탕으로 `디자인시스템` 문서를 분리하면 개발 핸드오프가 쉬워진다.
